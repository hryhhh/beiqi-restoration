# 壁画修复工作台设计

## 背景

当前系统已经具备壁画档案、病害标注、图像分析、修复方案和修复项目等页面能力，但缺少一个以单幅壁画为核心、面向修复生成结果的独立工作台。现有 `图像分析` 页面负责病害检测，不适合继续承载“上传壁画图像、调节修复参数、生成修复效果、确认保存”的修复闭环。

本次设计新增一个独立的“壁画修复工作台”页面，支持先选壁画，再上传待修复图像，调整默认与高级参数，执行整图修复或局部精修，在真实接口不可用时自动降级为演示模式，并在用户确认后将当前结果保存为该壁画的 `restored` 图层。

## 目标

- 提供一个独立的壁画修复工作台页面，不与病害检测页面混用。
- 页面以单幅壁画为核心，先选择系统内已有壁画，再上传本次待修复图像。
- 支持两种修复模式：整图修复、局部精修。
- 局部精修同时支持两条路径：选择已有病害标注、手动临时选区。
- 参数区采用双层模式：默认展示简洁参数，按需展开高级参数。
- 一次生成先输出一个主结果，支持基于当前参数继续生成变体。
- 结果先预览，用户满意后再保存为该壁画的 `restored` 图层。
- 修复接口优先走真实后端；真实接口不可用时自动降级为演示模式，但页面流程保持完整。

## 非目标

- 不把本次工作台直接并入 `图像分析` 页面。
- 不在第一版中接入修复方案审批流、项目流程流转或多人协作。
- 不做批量修复、多壁画并行处理、历史版本时间轴。
- 不引入复杂的专家级工艺参数体系，只保留能支撑第一版体验的核心参数。

## 方案比较

### 方案 A：在图像分析页扩展“检测 / 修复”双模式

优点：

- 可复用现有上传、预览和结果区逻辑，改动面较小。

缺点：

- 病害检测与修复生成职责混杂，页面会继续膨胀。
- 用户心智不清晰，后续迭代容易互相牵连。

### 方案 B：独立的“壁画修复工作台”页面

优点：

- 职责边界清晰，天然贴合“选壁画 -> 上传 -> 调参 -> 生成 -> 保存”的流程。
- 方便未来单独接入真实 AI 修复接口和更丰富的修复参数。
- 能复用现有壁画、标注、对比视图和图片上传能力，但不会污染原有页面。

缺点：

- 需要新增路由、菜单入口、API 封装和独立状态管理。

### 方案 C：在壁画详情页新增“修复工作台”标签

优点：

- 当前壁画上下文完整，保存回壁画图层最自然。

缺点：

- 与“先从工作台选壁画”的目标入口不一致。
- 壁画详情页已经较重，再塞入修复生成会让职责边界进一步模糊。

### 结论

采用 **方案 B：独立的“壁画修复工作台”页面**。页面入口独立，但实现上复用现有壁画列表、标注数据结构、图片对比组件和图片上传能力。

## 信息架构与页面入口

### 路由与导航

- 新增受保护页面路由：`/restoration`
- 侧边栏新增菜单项：`壁画修复`
- 后续可在壁画详情页补一个“前往修复工作台”的快捷入口，但这不是第一版主入口

### 页面布局

工作台采用用户确认过的 **B. 上下流程式** 布局，分为 5 个区块：

1. 顶部上下文区
2. 参数区
3. 原图区
4. 结果区
5. 底部操作区

布局目标是让用户从上到下完成一次完整修复流程，同时保留足够的可视化空间给原图、选区和结果图。

## 页面模块设计

### 1. 顶部上下文区

职责：

- 选择目标壁画
- 展示当前壁画基础信息
- 切换修复模式

包含内容：

- 壁画下拉选择器，来源于现有 `getMurals({ pageSize: 200 })`
- 当前壁画摘要卡片：名称、地点、时代、当前状态
- 修复模式切换：
  - `整图修复`
  - `局部精修`

行为约束：

- 未选择壁画时，上传区与修复按钮不可提交
- 切换壁画时，清空当前上传图、选区、结果和变体，避免数据串联

### 2. 参数区

参数区采用双层模式：

- 默认参数始终可见
- 高级参数默认折叠，按需展开

#### 默认参数

- `修复强度`
- `去污程度`
- `色彩还原`
- `细节保留`
- `裂隙修补倾向`

以上参数建议采用滑杆或分段控件，以降低第一次使用门槛。

#### 高级参数

按分组展示：

- `结构修补`
  - 裂隙闭合
  - 缺损填补
  - 边缘融合
- `表面清理`
  - 污渍清除
  - 霉斑抑制
  - 盐析弱化
- `色彩纹理`
  - 综合色偏修正
  - 局部色彩补全
  - 纹理重建强度
- `输出控制`
  - 清晰度优先 / 保真优先
  - 生成随机性
  - 变体继承当前参数继续生成

参数设计原则：

- 默认参数必须足够完成一次可用生成
- 高级参数只做增强，不应成为主流程阻碍
- 第一版参数命名面向业务人员，不直接暴露模型内部术语

### 3. 原图区

职责：

- 上传待修复原图
- 展示修复前图像
- 支撑整图修复或局部精修的选区操作

包含内容：

- 上传控件：支持点击上传和拖拽上传
- 原图预览区域
- 局部精修辅助区：
  - `已有标注` 选择面板
  - `手动选区` 入口

局部精修规则：

- 当模式为 `整图修复` 时，不要求选区
- 当模式为 `局部精修` 时，必须满足以下之一：
  - 选择至少一个已有病害标注
  - 完成一次手动选区

选区能力复用建议：

- 优先复用现有标注坐标结构 `AnnotationCoordinates`
- 手动选区尽量复用现有标注/画布交互能力，避免定义第二套坐标协议

### 4. 结果区

职责：

- 显示当前主结果图
- 显示变体缩略图
- 表达当前结果是否来自演示模式

结果区结构：

- 主结果视图：右侧主视觉区域，显示当前结果图
- 对比模式切换：
  - 并排前后对比
  - 滑块叠加对比
- 变体列表：显示基于当前参数继续生成的候选版本
- 结果状态标签：
  - `真实生成`
  - `演示模式`

交互原则：

- 第一次生成只突出一个“当前主结果”
- 变体放在次级区域，不与主结果争抢视觉焦点
- 用户切换变体时，只替换当前主结果，不打乱参数区

### 5. 底部操作区

职责：

- 驱动生成
- 再次生成变体
- 确认保存

核心操作：

- `开始修复`
- `再次生成变体`
- `确认保存为修复后图像`

行为规则：

- `开始修复` 在生成中进入 loading，避免重复提交
- `再次生成变体` 仅在已有主结果后可用
- `确认保存` 仅对当前选中的主结果生效

## 主流程设计

### 主流程

1. 用户进入 `/restoration`
2. 选择目标壁画
3. 上传待修复原图
4. 选择 `整图修复` 或 `局部精修`
5. 若为局部精修，则选择已有标注或手动画区
6. 调整默认参数，必要时展开高级参数
7. 点击 `开始修复`
8. 页面进入处理中状态，结果区显示进度反馈和等待态
9. 返回一张主结果图
10. 用户可点击 `再次生成变体`
11. 用户选定满意结果后点击 `确认保存`
12. 系统将当前结果保存为所选壁画的 `restored` 图层

### 变体流程

- 变体生成基于当前参数和当前主结果上下文继续生成
- 新变体进入缩略图列表
- 用户点击任意变体可将其切换为当前主结果

### 保存流程

- 保存前仅在页面内预览，不直接落库
- 用户确认保存后，把当前选中的结果图写回该壁画的 `restored` 图层
- 保存成功后，允许用户跳转到壁画详情页查看修复后图层

## 前后端接口策略

### 前端 API 分层

新增独立的 `restoration` API 封装，不并入 `analysis` 或 `plan`。

职责：

- 触发修复生成
- 基于当前结果生成变体
- 保存当前结果到壁画记录

### 建议接口 1：生成修复结果

建议新增后端接口，例如：

- `POST /api/restoration/generate`

请求内容：

- `muralId`
- `mode`：`full` 或 `partial`
- `sourceImage`
- `parameters`
- `annotationIds` 或 `manualSelection`
- `variantBaseId`（可选，用于基于当前结果继续生成变体）

返回内容：

- `resultId`
- `imageUrl`
- `mode`
- `isMock`
- `canGenerateVariant`
- `metadata`（可选，包含参数快照、耗时等）

### 建议接口 2：确认保存当前结果

第一版不强制新增“保存结果”专用后端接口，可采用以下实现策略：

- 前端获取当前结果图文件或结果 URL
- 复用现有壁画图片上传能力
- 以 `imageType=restored` 写入当前壁画图层

如果后端后续需要记录修复结果元数据，再新增专用保存接口。

## 真实接口优先、演示模式降级

第一版必须支持以下兼容策略：

- 前端优先调用真实修复接口
- 若接口不可用、超时或返回未实现，则自动降级到演示模式
- 演示模式下仍返回一个可预览的结果图，确保页面流程完整
- 结果区明确标记当前结果为 `演示模式`
- 即使是演示模式，用户仍可执行“确认保存”，但页面必须让其知道保存的是演示结果

## 与现有系统的衔接

### 与壁画模块的衔接

- 复用 `getMurals` 获取壁画列表
- 保存结果时写回壁画的 `restored` 图层
- 保存成功后，壁画详情页的图层资源和前后对比区可直接消费该结果

### 与标注模块的衔接

- 局部精修时可读取当前壁画已有标注
- 手动选区沿用现有坐标结构，减少协议分裂

### 与图像分析模块的边界

- `analysis` 继续负责病害检测与转标注
- `restoration` 只负责修复生成与结果保存
- 两者共享壁画与标注上下文，但不共用同一个页面入口

### 与修复方案模块的边界

- 第一版不把“生成结果”直接挂入修复方案审批流
- 后续如需联动方案审批，再单独扩展结果元数据与关系表

## 状态与数据模型建议

前端页面至少需要维护以下状态：

- 当前选中的壁画
- 当前上传原图
- 当前修复模式
- 默认参数与高级参数
- 已选标注 ID 列表
- 手动选区坐标
- 当前主结果
- 变体列表
- 是否为演示模式
- 生成中、保存中状态

建议结果对象结构至少包含：

- `id`
- `imageUrl`
- `isMock`
- `createdAt`
- `parametersSnapshot`
- `sourceType`（主结果或变体）

## 异常处理

第一版需要明确处理以下异常：

- 未选壁画即尝试上传或修复
- 未上传原图即尝试开始修复
- 局部精修未选择标注且未完成手动选区
- 真实接口不可用或生成失败
- 生成过程中重复点击
- 保存失败
- 切换壁画后旧结果残留

处理原则：

- 能阻止的前置条件在前端直接阻止
- 真实接口失败时自动降级而不是直接报错中断
- 保存失败时保留当前结果和参数，避免用户损失本次工作成果

## 测试范围

第一版验证重点围绕完整闭环，而不是细节穷举。

需要覆盖的核心路径：

- 选择壁画 -> 上传原图 -> 整图修复 -> 保存
- 选择壁画 -> 上传原图 -> 局部精修（已有标注） -> 保存
- 选择壁画 -> 上传原图 -> 局部精修（手动选区） -> 保存
- 默认参数提交与高级参数提交
- 真实接口失败 -> 自动切换演示模式 -> 仍可预览和保存
- 生成主结果后继续生成变体，并切换当前主结果

## 第一版范围控制

第一版包含：

- 独立的壁画修复工作台页面
- 上下流程式布局
- 整图修复与局部精修
- 已有标注与手动选区
- 默认参数与高级参数
- 单主结果 + 再生成变体
- 结果确认保存到 `restored` 图层
- 真实接口优先、失败自动降级 mock

第一版不包含：

- 修复方案审批联动
- 多人协同会话
- 历史版本时间轴
- 批量修复
- 复杂专家参数体系

## 风险与后续扩展

### 第一版风险

- 真实修复接口尚不存在，前端需先按可扩展协议设计并承担降级逻辑
- 局部精修若直接复用现有标注画布，需控制交互复杂度，避免把标注页的全部能力搬进来
- 结果保存若完全复用壁画图片上传，需要明确结果图数据来源与格式转换方案

### 后续扩展方向

- 将修复结果与修复方案、审批记录建立关联
- 为结果补充版本历史与时间轴
- 记录每次修复的参数快照、生成耗时和备注
- 引入更专业的材质、病害类型驱动参数模板
